# Security Model

MorScan authenticates three kinds of caller:

1. **Public endpoints** - a small allowlist (`src/routes/public.ts`), e.g.
   `GET /mor/v1/price`, `/health`, `/openapi.json`, plus every read-only
   explorer page. No auth.
2. **Wallet-signed SDK traffic** - gated `/mor/v1/*` endpoints accept
   EIP-712 wallet-signed requests (signature headers verified in
   `src/utils/wallet-auth.ts`, derived-signer attestation in D1).
3. **API keys** - `X-Morscan-Key` (or `Authorization: Bearer`) for direct API
   callers, personal keys from `/console`, the UI serving key, and admin
   (`/sync/*`) callers.

## API key gating

Keys are stored in the D1 `api_keys` table, format `mor_<random>`. The
env-configured serving key is compared with a timing-safe equality check.

| Key | Purpose | Limits |
|-----|---------|--------|
| Admin | Generated by the operator (see [`deployment.md`](deployment.md)); the row with id `admin` (plus any ids in `MORSCAN_ADMIN_KEY_IDS`). Required for `/sync/*` admin routes. | Exempt |
| Per-wallet | Minted free on wallet connect at `/console` (row id `wallet:<address>`). Capacity follows the wallet's live MOR stake on the MorScan builder subnet: unstaked keys get the bottom tier (10/min, 2,000/day, 40,000/month); staking raises the caps, re-checked every minute (`src/utils/stake-tier.ts`, [`rate-limiting.md`](rate-limiting.md)). | Stake-indexed |
| Serving key | `MORSCAN_DEMO_KEY` secret, injected into public pages as `window.MORSCAN_API_KEY` so page scripts can call the metered API. Read-only; rejected from admin routes. | High per-key aggregate; every client IP still bound by the per-IP limit |

Admin sync routes (`/trigger-sync`, `/sync/*`) check that the authenticated
key's `keyId` is an admin id; any other key (including the serving key)
receives a 403. The serving key returns a distinct `keyId` (`demo`) so it can
never masquerade as admin in logs or authorization checks.

## Wallet sign-in (`/console`)

The console is the site's single sign-in door:

- `GET /console/wallet/challenge` mints a single-use nonce (KV, 5 min TTL).
- `POST /console/wallet/verify` recovers the `personal_sign` signer; on match it
  creates the session and auto-issues the wallet's free API key if absent.
- The nonce is deleted on first use (no replay).

## Rate limiting

Two layers on every gated request; either can reject.

- **Minute burst**: per-IP (100/min) + per-key (`api_keys.rate_limit`), enforced
  via Workers rate-limiting bindings when configured, with a per-isolate
  in-memory fallback.
- **Day/month volume**: `usage_counters` rows per key, caps from the key's
  `daily_cap`/`monthly_cap` columns.
- **Sign-in brute-force**: `POST /login` and the wallet challenge/verify routes
  are limited per IP.

Implementation: `src/utils/auth/` - `validateApiKey()`, `checkRateLimit()`,
timing-safe key comparison. See [`rate-limiting.md`](rate-limiting.md).

## JWT sessions (console only)

- Wallet verify (or `POST /login` with a valid key, for returning key holders)
  issues a JWT in an HttpOnly cookie.
- The JWT is signed with `MORSCAN_JWT_SECRET` (a Worker secret, **required** -
  session creation fails without it; no default = no auth bypass).
- Cookie: HttpOnly, Secure, SameSite=Lax, 24h expiry.
- Open-redirect protection: `safeRedirect()` only accepts relative-path return
  URLs (must start with `/`, must not start with `//`).

Implementation: `src/utils/jwt.ts`, `src/utils/auth/`, `src/routes/auth/`.

## Input validation

- Address path params validated with strict regex `0x[0-9a-fA-F]{40}` before
  reaching handlers; pagination validated with `Number.isFinite()`.
- ABI parse functions validate response bounds before slicing; batch RPC
  responses are matched by an id `Map` (not sort order) to prevent
  misattribution by a malicious RPC.
- Event log topics are length-checked before indexed access.

## Output security

- HTML responses set `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`,
  a `Content-Security-Policy`, and `Referrer-Policy: strict-origin-when-cross-origin`.
- All blockchain data is escaped before rendering (`escHtml()`, `escJs()`,
  `safeJson()` for embedded JSON).
- Error responses return a generic message; full details are logged server-side
  only (no table names, RPC URLs, or stack traces to clients).

## CORS

`Access-Control-Allow-Origin: *` is the default so the read API is broadly
consumable. Restrict it to specific origins if you gate writes behind cookies.

## No localStorage

Zero client state on disk. Auth is server-controlled JWT cookies; page state
resets on load.
